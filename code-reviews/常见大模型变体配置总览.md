# 常见大模型变体配置总览

## 1. 变体列表总览

| 变体 ID | 目标模型 | 工具格式 | 特点 |
|---------|----------|----------|------|
| `generic` | 通用/后备 | XML 文本 | 最完整的组件，作为后备方案 |
| `next-gen` | Claude 4.5 等 | XML 文本 | 丰富规则，前沿模型优化 |
| `native-next-gen` | Claude 4.5 (原生) | Function Calling | API 原生工具调用 |
| `glm` | 智谱 GLM-4.6 | XML 文本 | 智谱特定优化，简化规则 |
| `xs` | 本地小模型 | XML 文本 | 极度精简，省 token |
| `gpt-5` | GPT-5 系列 | XML 文本 | OpenAI 特定优化 |
| `native-gpt-5` | GPT-5 (原生) | Function Calling | OpenAI 原生工具 |
| `gemini-3` | Gemini 3.0 | Function Calling | Google 原生工具 |
| `trinity` | Trinity 模型 | XML 文本 | 特定模型优化 |

## 2. 模板结构规范

基础模板使用占位符语法 `{{PLACEHOLDER}}`，由 `====` 分隔各部分：

## 3. 组件（Section）说明

| 组件 ID | 内容 | 作用 |
|---------|------|------|
| `AGENT_ROLE` | "You are Cline, a highly skilled software engineer..." | 定义 AI 的身份和角色 |
| `TOOL_USE` | 工具使用说明和格式 | 告诉 AI 如何调用工具 |
| `TASK_PROGRESS` | 任务进度和 TODO 列表 | 当前任务状态 |
| `MCP` | MCP 服务器列表和使用方式 | 外部工具扩展 |
| `EDITING_FILES` | 文件编辑规范 | 如何正确编辑文件 |
| `ACT_VS_PLAN` | Plan/Act 模式说明 | 计划模式 vs 执行模式 |
| `CAPABILITIES` | AI 能力描述 | 可以做什么 |
| `SKILLS` | 可用技能列表 | 可调用的技能 |
| `RULES` | 行为规则 | 必须遵守的规则 |
| `SYSTEM_INFO` | 系统信息（OS、Shell 等） | 运行环境 |
| `OBJECTIVE` | 当前任务目标 | 用户的请求 |
| `USER_INSTRUCTIONS` | 用户自定义指令 | .clinerules 内容 |

## 4. 变体配置 Builder 模式

```typescript
createVariant(ModelFamily.NEXT_GEN)
  // 1. 基本信息
  .description("Prompt tailored to newer frontier models")
  .version(1)
  .tags("next-gen", "advanced", "production")
  .labels({ stable: 1, production: 1 })
  
  // 2. 匹配器 - 决定何时使用此变体
  .matcher((context) => {
    return isNextGenModelFamily(context.providerInfo.model.id)
  })
  
  // 3. 模板 - 提示词骨架
  .template(baseTemplate)
  
  // 4. 组件顺序 - 要包含哪些部分
  .components(
    SystemPromptSection.AGENT_ROLE,
    SystemPromptSection.TOOL_USE,
    // ...
  )
  
  // 5. 工具列表 - 该变体可用的工具
  .tools(
    ClineDefaultTool.BASH,
    ClineDefaultTool.FILE_READ,
    // ...
  )
  
  // 6. 占位符值
  .placeholders({ MODEL_FAMILY: ModelFamily.NEXT_GEN })
  
  // 7. 组件覆盖 - 定制特定组件
  .overrideComponent(SystemPromptSection.RULES, {
    template: rules_template,
  })
  
  .build()
```

## 5. 不同变体的差异对比

| 特性 | generic | next-gen | xs (小模型) | native-* |
|------|---------|----------|-------------|----------|
| 工具格式 | XML | XML | XML | 原生 Function Calling |
| 组件数量 | 完整 | 完整 | 精简 | 精简 |
| 规则详细度 | 高 | 很高 | 低 | 中 |
| MCP 支持 | ✅ | ✅ | ❌ | 部分 |
| 浏览器工具 | ✅ | ✅ | ❌ | ✅ |
| TODO 管理 | ✅ | ✅ | ❌ | ✅ |
| 子代理 | ✅ | ✅ | ✅ | ✅ |

## 6. 匹配优先级

当调用 `PromptRegistry.get(context)` 时，按以下顺序匹配：

```
1. native-next-gen   (enableNativeToolCalls && isNextGenModel)
2. native-gpt-5      (enableNativeToolCalls && isGPT5Model)
3. next-gen          (isNextGenModel)
4. gpt-5             (isGPT5Model)
5. glm               (isGLMModel)
6. xs                (isLocalModel && compact)
7. gemini-3          (isGeminiModel)
8. trinity           (isTrinityModel)
9. generic           (fallback - 所有其他情况)
```

## 8. 各变体详细规范

### 8.1 Generic（通用变体）

**用途**：所有未匹配其他变体的模型的后备方案

**匹配条件**：
```typescript
matcher: (context) => {
  return (
    !(customPrompt === "compact" && isLocalModel) &&
    !isNextGenModelFamily(modelId) &&
    !isGLMModelFamily(modelId) &&
    !isTrinityModelFamily(modelId)
  )
}
```

**模板结构**（使用 `====` 分隔符）：
```
AGENT_ROLE_SECTION
TOOL_USE_SECTION
====
TASK_PROGRESS_SECTION
====
MCP_SECTION
====
EDITING_FILES_SECTION
====
ACT_VS_PLAN_SECTION
====
CAPABILITIES_SECTION
====
SKILLS_SECTION
====
FEEDBACK_SECTION
====
RULES_SECTION
====
SYSTEM_INFO_SECTION
====
OBJECTIVE_SECTION
====
USER_INSTRUCTIONS_SECTION
```

**工具列表**（完整）：
- `execute_command`, `read_file`, `write_to_file`, `replace_in_file`
- `search_files`, `list_files`, `list_code_definition_names`
- `browser_action`, `use_mcp_tool`, `access_mcp_resource`
- `ask_followup_question`, `attempt_completion`, `plan_mode_respond`
- `load_mcp_documentation`, `cline_todo`, `generate_explanation`
- `use_skill`, `use_subagents`

---

### 8.2 Next-Gen（前沿模型变体）

**用途**：Claude 4.5 Sonnet/Opus 等前沿模型，使用 XML 文本工具格式

**匹配条件**：
```typescript
matcher: (context) => {
  return (
    isNextGenModelFamily(modelId) &&
    !context.enableNativeToolCalls &&
    !(customPrompt === "compact" && isLocalModel)
  )
}
```

**特点**：
- 完整的规则集，更详细的行为规范
- 支持 Web 工具（`web_fetch`, `web_search`）
- 详细的 RULES 组件覆盖

**RULES 组件覆盖**（部分示例）：
```typescript
// 工作目录规则
- Your current working directory is: {{CWD}}
- You cannot cd into a different directory

// 工具使用规则
- When using replace_in_file, include complete lines in SEARCH blocks
- List SEARCH/REPLACE blocks in the order they appear in the file

// 行为规则
- NEVER start messages with "Great", "Certainly", "Okay", "Sure"
- NEVER end attempt_completion with a question
```

---

### 8.3 Native-Next-Gen（原生工具前沿变体）

**用途**：支持 API 原生 Function Calling 的前沿模型

**匹配条件**：
```typescript
matcher: (context) => {
  return (
    context.enableNativeToolCalls &&
    isNextGenModelProvider(providerInfo) &&
    isNextGenModelFamily(modelId) &&
    !isGPT5ModelFamily(modelId)
  )
}
```

**特点**：
- 使用 API 原生工具调用格式（非 XML）
- 精简的 TOOL_USE 说明
- 支持并行工具调用

**TOOL_USE 组件覆盖**：
```typescript
const TOOL_USE = (context) => `TOOL USE

You have access to a set of tools that are executed upon the user's approval.${
  context.enableParallelToolCalling 
    ? " You may use multiple tools in a single response when independent." 
    : ""
}`
```

**RULES 组件覆盖**（简化版）：
```typescript
const RULES = (context) => `RULES

- The current working directory is \`{{CWD}}\`
${context.enableParallelToolCalling ? 
  "- You may use multiple tools when operations are independent." : ""
}`
```

### 8.4 GLM（智谱模型变体）

**用途**：智谱 GLM-4.6 模型专用优化

**匹配条件**：
```typescript
matcher: (context) => isGLMModelFamily(context.providerInfo.model.id)
```

**模板结构**（使用 `##` 标题分隔）：
```
AGENT_ROLE_SECTION
TOOL_USE_SECTION
## TASK_PROGRESS_SECTION
## RULES_SECTION
## ACT_VS_PLAN_SECTION
## CAPABILITIES_SECTION
## SKILLS_SECTION
## EDITING_FILES_SECTION
## TODO_SECTION
## MCP_SECTION
## SYSTEM_INFO_SECTION
## OBJECTIVE_SECTION
## USER_INSTRUCTIONS_SECTION
```

**组件覆盖**：

1. **TOOL_USE**（完全自定义）：
```
Begin every task by exploring the codebase (list_files, search_files, read_file) 
and outlining the required changes. Do not implement until exploration yields 
enough context to state objectives, approach, affected files, and risks.

Tool invocation policy: Invoke tools only in assistant messages; 
they will not execute if placed inside reasoning blocks.

## TOOLS
**execute_command** — Run terminal commands in {{CWD}}
**read_file** — Read file. Params: path.
**write_to_file** — Create/overwrite file (new files only)
**replace_in_file** — Targeted edits (existing files)
...
```

2. **RULES**（精简版）：
```
- Accomplish the user's task; avoid back-and-forth conversation.
- Your working directory is {{CWD}}. You cannot cd elsewhere.
- If command output doesn't appear, assume success and continue.
- After each tool use, wait for user's response to confirm success.
```

3. **TASK_PROGRESS**（带示例）：
```
Each tool supports an optional task_progress parameter for maintaining 
a Markdown checklist of your progress.

Example:
<execute_command>
<command>npm install react</command>
<task_progress>
- [x] Set up project structure
- [x] Install dependencies
- [ ] Create components
</task_progress>
</execute_command>
```

### 8.5 XS（小模型精简变体）

**用途**：本地小上下文窗口模型，极度精简以节省 token

**匹配条件**：
```typescript
matcher: (context) => {
  return isLocalModel(providerInfo) && customPrompt === "compact"
}
```

**模板结构**（最精简）：
```
AGENT_ROLE_SECTION
## RULES_SECTION
## ACT_VS_PLAN_SECTION
## CAPABILITIES_SECTION
## SKILLS_SECTION
## EDITING_FILES_SECTION
## TOOL_USE_SECTION
## OBJECTIVE_SECTION
## SYSTEM_INFO_SECTION
## USER_INSTRUCTIONS_SECTION
```

**工具列表**（精简）：
- `execute_command`, `read_file`, `write_to_file`, `replace_in_file`
- `search_files`, `ask_followup_question`, `attempt_completion`
- `plan_mode_respond`, `use_subagents`
- ❌ 无 MCP、无浏览器、无 TODO

**组件覆盖**：

1. **AGENT_ROLE**（一句话）：
```
You are Cline, a senior software engineer + precise task runner. 
Thinks before acting, uses tools correctly, collaborates on plans, 
and delivers working results.
```

2. **RULES**（极简）：
```
GLOBAL RULES
- One tool per message; wait for result. Never assume outcomes.
- Exact XML tags for tool + params.
- CWD fixed: {{CWD}}; to run elsewhere: cd /path && cmd in one command
- Edits: replace_in_file default; exact markers; complete lines only.
- Tone: direct, technical, concise.
```

3. **ACT_VS_PLAN**（紧凑）：
```
MODES (STRICT)
**PLAN MODE (read-only):**
- Allowed: plan_mode_respond, read_file, list_files, search_files...
- Do NOT run CLI, create/modify files, or call execute_command

**ACT MODE:**
- Allowed: all tools except plan_mode_respond
- Implement stepwise; one tool per message
```

4. **TOOLS**（简短示例）：
```
**execute_command** — Run CLI in {{CWD}}. Params: command, requires_approval.
<execute_command>
<command>npm run build</command>
<requires_approval>false</requires_approval>
</execute_command>

**read_file** — Read file. Param: path.
<read_file><path>src/App.tsx</path></read_file>
```

## 10. 如何添加新变体

1. 在 `src/core/prompts/system-prompt/variants/` 下创建新目录
2. 创建 `config.ts`，使用 `createVariant()` 构建配置
3. 创建 `template.ts`，定义基础模板
4. 可选：创建 `overrides.ts`，定制组件内容
5. 在 `src/core/prompts/system-prompt/variants/index.ts` 中导出新变体

### 示例：添加新模型变体

```typescript
// variants/my-model/config.ts
import { ModelFamily } from "@/shared/prompts"
import { ClineDefaultTool } from "@/shared/tools"
import { SystemPromptSection } from "../../templates/placeholders"
import { createVariant } from "../variant-builder"
import { baseTemplate } from "./template"

export const config = createVariant(ModelFamily.MY_MODEL)
  .description("Prompt optimized for MyModel")
  .version(1)
  .tags("my-model", "stable")
  .labels({ stable: 1 })
  
  // 匹配器：决定何时使用此变体
  .matcher((context) => {
    const modelId = context.providerInfo.model.id.toLowerCase()
    return modelId.includes("my-model")
  })
  
  // 模板
  .template(baseTemplate)
  
  // 组件顺序
  .components(
    SystemPromptSection.AGENT_ROLE,
    SystemPromptSection.TOOL_USE,
    SystemPromptSection.RULES,
    // ...
  )
  
  // 工具列表
  .tools(
    ClineDefaultTool.BASH,
    ClineDefaultTool.FILE_READ,
    // ...
  )
  
  // 占位符
  .placeholders({ MODEL_FAMILY: "my-model" })
  
  // 可选：覆盖特定组件
  .overrideComponent(SystemPromptSection.RULES, {
    template: myCustomRulesTemplate,
  })
  
  .build()
```
